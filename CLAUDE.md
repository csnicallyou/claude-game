Контекст для всех будущих сессий Claude Code в этом проекте. Этот файл — единственный источник истины по «как писать код» и «что мы строим». Читать в начале каждой сессии.

> **Рабочее название:** Epochs of Humanity (англ.) / Эпохи Человечества (рус.).
> **Slug проекта:** `claude-game` (имя репозитория и папки).
> **Дата фиксации спецификации:** 2026-05-17.

---

## 1. TL;DR

Гранд-стратегия в духе Paradox (CK3 + EU4) с глубокой симуляцией поселения в духе Songs of Syx, эволюция от верхнего палеолита (~45,000 BCE) до современности и далее. Игрок — вождь племени (CK3-стиль наследования), смерть = переход к наследнику. Эпохи добавляются модулями. Стек: **Godot 4.3+ с C# (.NET 8)**. Solo-разработка, Claude — основной программист.

---

## 2. Vision

Игрок начинает на Леванте около 45,000 лет до н.э. за конкретного вождя малой группы Homo sapiens, недавно вышедшей из Африки. В окрестностях — другие группы сапиенсов и племена неандертальцев. Мегафауна, реки, горы Кармель.

Игра разворачивается на трёх масштабах одновременно:

- **Стратегический.** Гексагональная карта Леванта с биомами, провинциями, соседними племенами. Игрок ставит миграции, дипломатические цели, культурные направления. Темп — годы за секунды.
- **Тактический.** Зум в собственную стоянку: top-down пиксель-арт вид сверху, отдельные жители (до 500 на тактической карте), назначение охоты, ремесла, ритуалов. Темп — часы/сутки за секунды.
- **Личный.** Жизнь действующего вождя: важные решения (брак, болезнь, видение, конфликт с шаманом) появляются как модальные события в стиле CK3. При смерти вождя — переход в наследника.

Над объективной симуляцией (климат, биология, экономика) лежит **слой восприятия**: персонажи видят мир через анимистическую/шаманскую/тотемическую линзу палеолита. Засуха — это «дух реки разгневан», даже если механически это атмосферная аномалия. Эта линза **эволюционирует с эпохами** (анимизм → агрокульты → политеизм → монотеизм → идеологии Нового времени).

Род игрока ведёт автоматическую летопись — текстовую/визуальную хронику тысячелетий, которая и есть основной «нарратив» долгой игры.

**Game over:** род пресёкся (soft, можно начать за дочернее племя), племя физически вымерло (hard), племя поглощено соседом (продолжаем играть за вассальную линию).

**MVP-горизонт:** 6-9 месяцев при 20+ часах в неделю. MVP покрывает только палеолит и только Левант.

## 4. Архитектура — Четыре закона

### Заметка о глобальной карте (post-v1.0)

Когда модули принесут регионы за пределами Леванта (Месопотамия, Европа, Африка, Сибирь, Сахул и т.д.), **мы НЕ делаем единую гекс-сетку всей Земли**. Причины:

- На сфере нет идеального гекс-разбиения (Goldberg-полиэдр требует пентагонов на стыках).
- ~200,000 гексов на всю Землю при текущем рендере (`Polygon2D` на гекс) = неюзабельно.

**Правильный паттерн:** каждый регион — отдельный плоский гекс-патч со своими axial-координатами. Регионы регистрируются как **модули регионов** (аналогично модулям эпох): `IRegionPreset` (появится в v0.5+), `LevantPreset` → его первая реализация. Глобальная карта мира (когда понадобится) — **отдельная сцена**: стилизованная Земля с маркерами регионов, клик → зум в гекс-патч региона. Паттерн Total War (campaign vs battle) и CK3 (world map + provinces).

Cross-region взаимодействия (миграции, торговля, диффузия технологий) — на абстрактном уровне через события, не на единой сетке. Это согласуется с двухслойной симуляцией (Закон + блок 3.3): внутри активного региона — индивиды, во внешнем мире — попы.



Эти законы — **неприкосновенны** без явного согласования с пользователем. Нарушение любого из них ломает проект в долгосрок.

### Закон 1: Детерминизм

Если те же входы и сид — тот же результат, на любом железе, в любой версии .NET, в любой ОС. Это нужно для сейвов, replay, мультиплеера, отладки.

- Один `Rng` на мир, сидируется при создании / при загрузке сейва.
- Каждая подсистема получает дочерний PRNG через `rng.Fork("subsystem-name")`. Изменения в одной подсистеме не должны сдвигать случайности в другой.
- **Запрещено в `core/*`, `sim/*`, `eras/*`:**
  - `System.Random` (используем кастомный `Rng`)
  - `DateTime.Now`, `DateTime.UtcNow`, `Environment.TickCount`, `Stopwatch` — для логики симуляции. В рендере и профилировании — разрешено.
  - `Godot.GD.Randf()`, `Godot.RandomNumberGenerator` без сидирования и форка.
  - Итерация по `Dictionary<K, V>` без явной сортировки ключей.
  - Итерация по `HashSet<T>` для логики, влияющей на состояние.
- **Числа с плавающей точкой в симуляции:** допускаются только `double`, никаких `float`. Для финансово-критичной механики (бой, наследование, голод) — рассмотрим fixed-point в будущем. В MVP — `double` с осознанием рисков.

### Закон 2: Разделение Sim ↔ Render

Симуляция (`core/`, `sim/`, `eras/`) — **чистый C# без зависимостей от Godot.** Это значит:

- **Нельзя:** `using Godot;` в файлах `core/`, `sim/`, `eras/`.
- **Нельзя:** наследоваться от `Node`, `Resource` и других Godot-типов в симуляционных классах.
- **Можно:** в симуляции использовать `System.Numerics.Vector2/Vector3`, `Math`, `MathF`.

Рендер (`render/`, `ui/`) читает срез состояния через явный `WorldView` API и НЕ пишет в состояние. Любое действие игрока → создание `Command`-объекта → передача в очередь команд → исполнение симуляцией на следующем тике.

Польза: симуляцию можно прогонять headless (для тестов, для серверной части мультиплеера в будущем, для replay-анализа). Можно заменить рендер целиком, не трогая sim.

### Закон 3: Эпоха = Модуль

Ядро (`core/`) и общая симуляция (`sim/`) **не знают** о существовании конкретных эпох. Эпохи (`eras/<id>/`) импортируют только публичные API ядра и общей симуляции.

```csharp
// eras/paleolithic/PaleolithicModule.cs
public sealed class PaleolithicModule : IEraModule
{
    public string Id => "paleolithic";
    public string DisplayName => "Палеолит";
    public TimeRange TimeRange => new(startYearBP: 45_000, endYearBP: 10_000);

    public void Init(IEraInitContext ctx) { /* регистрация технологий, биомов, событий */ }
    public void Tick(IEraTickContext ctx) { /* per-tick logic */ }
    public void Content(IEraContentContext ctx) { /* контент из data/eras/paleolithic/ */ }
    public void Rehydrate(IEraRehydrateContext ctx, EraSnapshot snapshot) { /* при загрузке сейва */ }
}
```

Контексты дают эпохе доступ только к разрешённому подмножеству API. Передавать весь `World` напрямую — антипаттерн.

**Запрещённые направления import:**

- `core → sim / render / ui / eras` (никогда)
- `sim → render / ui / eras` (никогда)
- `eras/A → eras/B` (эпохи не знают друг о друге)
- `data → *` (data — это данные, не код)

### Закон 4: Dual-Reality (Симуляция ↔ Восприятие)

Над **слоем симуляции** (объективная реальность: климат, биология, экономика, демография) лежит **слой восприятия**: каждый персонаж интерпретирует события через когнитивно-культурную линзу эпохи и культуры.

```csharp
// core/perception/IWorldview.cs
public interface IWorldview
{
    string Id { get; }                                              // "paleolithic-animism"
    EraId ApplicableEra { get; }
    PerceivedEvent Interpret(IObjectiveEvent objectiveEvent,        // что случилось
                              ICharacterContext character);          // кто это видит
}

// core/perception/PerceivedEvent.cs
public sealed record PerceivedEvent(
    string ObjectiveEventId,          // ссылка на объективное событие
    string PerceivedCauseId,          // как персонаж это объясняет ("river-spirit-angry")
    EmotionalResponse Emotion,        // что чувствует
    ActionSuggestion[] Suggestions);  // какие действия персонаж считает уместными
```

**Инвариант:** слой симуляции и слой восприятия — два отдельных API. Симуляция не «знает», что в голове у персонажей. Восприятие читает срез симуляции и накладывает на него свою линзу.

**Эволюция мировоззрений по эпохам (commitment):**

| Эпоха | Worldview ID | Что добавляется механически |
|---|---|---|
| Палеолит (MVP) | `paleolithic-animism` | Анимизм, шаманизм, тотемизм, культ предков, табу |
| Мезолит | `mesolithic-ancestor-cult` | Сакральные места, культ предков-героев |
| Неолит | `neolithic-fertility` | Аграрные культы, богини-матери, мегалиты |
| Бронза | `bronze-pantheon` | Пантеон, царь-жрец, кодифицированная мифология |
| Античность | `antiquity-statereligion` | Государственные религии, философия |
| Средневековье | `medieval-monotheism` | Монотеизм, теология, ереси |
| Новое время | `modern-secularism` | Наука vs религия, секуляризация |
| Современность | `contemporary-ideology` | Идеологии (национализм, либерализм, etc.) |

Каждый Worldview регистрируется через тот же `IEraModule.Content()` API, что и остальной контент эпохи.

## 7. Конвенции

### Именование

- **Файлы C#:** PascalCase, по имени основного типа (`GameLoop.cs`, `PaleolithicModule.cs`)
- **Папки:** kebab-case (`game-loop`, `data-loader`)
- **Сцены Godot (.tscn):** PascalCase (`MainMenu.tscn`)
- **Конфиги JSON:** kebab-case (`tech-tree-paleolithic.json`)
- **Локализационные ключи:** dot.case (`ui.menu.new_game`, `event.shaman_vision.title`)
- **Имена в коде:** C# конвенция (PascalCase для типов/методов/публичных полей, camelCase для локальных, `_camelCase` для приватных полей)
- **Археологические имена** (Карьмель, Кебара, Кафзе, Сапиенс, Неандерталец) — **на латинице во всех языках**, как реальные термины: `Carmel`, `Qafzeh`, `Skhul`, `Kebara`, `Sapiens`, `Neanderthal`. Это и реализм, и облегчает локализацию.

### Стиль кода

- **Без `dynamic`, без `object` без необходимости.** Используем generics и `nullable reference types` (`<Nullable>enable</Nullable>` в .csproj).
- **`readonly`/`record`/`sealed` по умолчанию.** Иммутабельность снижает баги. `class` без `sealed` — только если нужен полиморфизм.
- **Без статичного state в симуляции.** Всё через явный `World`-контекст.
- **Один файл — одна осмысленная единица.** Файл > 300 строк → подумать о разбиении.
- **Тесты рядом с кодом** в `tests/`, mirror структуру: `core/loop/GameLoop.cs` → `tests/core/loop/GameLoopTests.cs`.
- **Игровые числа — в конфигах**, не хардкод. Магические числа в логике запрещены, кроме общеизвестных констант (π, секунд в минуте).
- **Комментарии — только для «почему»**, не для «что» (см. global CLAUDE.md).
- **XML-doc** на публичных API ядра и общих симуляционных утилитах. На приватных — не обязательно.

### Перед коммитом

1. `dotnet build` — без warnings (включаем `<TreatWarningsAsErrors>true</TreatWarningsAsErrors>` в csproj).
2. `dotnet test` — все тесты зелёные.
3. `dotnet format` — отформатировано через `.editorconfig`.
4. Godot открывается без ошибок в редакторе.

### Git

- Формат коммита: `тип: описание` (feat/fix/docs/chore/refactor/test/style/perf) — см. global CLAUDE.md.
- Один коммит = одна семантическая правка.
- Не push в `main` форс. Не amend опубликованного. Не `--no-verify`.

## 8. Что НЕ делать без явного согласования с пользователем

- Устанавливать новые NuGet-пакеты (через `dotnet add package`).
- Устанавливать новые Godot-плагины.
- Менять API ядра (`core/*`) — это сломает все эпохи. Расширять — можно, ломать — нет.
- Использовать `System.Random`, `DateTime.Now`, `Stopwatch` в `core/sim/eras/*` (см. Закон 1).
- Использовать `using Godot;` в `core/sim/eras/*` (см. Закон 2).
- Создавать новые папки верхнего уровня вне согласованной задачи.
- Переписывать существующие эпохи целиком при работе над одной.
- Запускать миграции сейвов до v0.5 (мы их не делаем).
- Force-push, амендить опубликованные коммиты, бэйпасить хуки.
- Читать `obsidian/notes/personal/` или `journal/` (см. global CLAUDE.md).

## 9. Roadmap

Версии — semver-подобные, но без формального гарантии стабильности до v1.0.

### v0.1 — «Карта Леванта существует» (~1 мес от старта проекта)

- Godot-проект собирается, C# компилируется.
- Гексагональная карта Леванта (~200-500 гексов), биомы, статичная.
- 4-5 NPC-племён на карте, не двигаются, но видны.
- Один игрок-вождь с именем и возрастом. Можно «прожить год» (нажать «дальше»).
- Сейв/загрузка работают (без миграций).
- RU/EN локализация интерфейса.
- Все четыре закона архитектуры соблюдены.

### v0.2 — «Симуляция идёт» (~2 мес)

- Game loop с фиксированным шагом + 4 скорости.
- Племена двигаются, размножаются, умирают (попы).
- Сезоны меняются, температура меняется, ресурсы обновляются.
- Вождь стареет, женится, имеет детей, умирает → наследование.
- Базовый AI NPC-племён (миграции по голоду).
- Минимальные SFX.

### v0.3 — «Тактический слой появляется» (~3 мес)

- Зум в собственное поселение: top-down карта стоянки.
- ~50-100 индивидуумов в собственной стоянке (не попы).
- Назначение работ: охота, собирательство, ремесло, ритуал.
- Простой бой между группами.

### v0.4 — «Восприятие включено» (~4 мес)

- Слой восприятия (Закон 4) — анимизм, шаманизм, табу.
- События в стиле CK3 с интерпретацией через мировоззрение.
- Шаман как роль с механиками видений.

### v0.5 — «Дипломатия и миграции» (~5 мес)

- Полноценная дипломатия между племенами (союзы, обмен, война).
- Миграция через карту (например, в условиях изменения климата).
- **Включаем save migrations**, схема сейва стабилизируется.

### v0.6 — «Полировка и культурный слой» (~6 мес)

- Технологии как **Tradition Tree** (Pillar 3): передаются между поколениями, могут забываться при разрыве трансмиссии.
- Культурные различия между племенами (язык, обряды, искусство).
- **Летопись рода** (Pillar 1) — автогенерируемая текстово-визуальная хроника, главный shareable артефакт.
- **Memory of Place** (Pillar 4) — хексы накапливают «памятный вес» от событий, генерируют сакральные места.
- **Slow Apocalypse arc** (Pillar 5) — первое крупное палеоклиматическое событие (Heinrich Event 4 ~38k BCE) как сценарная арка.

### v0.7 — «MVP playable» (~7-9 мес)

- Все системы стабильны, баланс играбелен.
- Полная RU/EN локализация, включая внутриигровой контент.
- Открытый репозиторий, README, contribution guide.
- Возможный запуск closed alpha для тестеров.

### v0.8 — Polish + Steam EA prep

- UX-полировка, accessibility, контроллер-поддержка.
- Возможный Steam Early Access.

### v0.9 — Unique pillars expansion

- **Asymmetric species mechanics** (Pillar 2): полная асимметрия Sapiens / Neanderthal в навыках, размере групп, культурной передаче. Гибриды как механика.
- **Cognitive variants** (Pillar 6): редкие когнитивные стили у персонажей.
- **Procedural archeology** (Pillar 9): заброшенные стоянки оставляют находки для будущих поколений.

### v1.0 — «Палеолит готов»

- Полностью отполированный палеолит как самодостаточная игра.
- Архитектурные хуки готовы для модулей следующих эпох.
- Mod API для конфигов задокументирован.
- **Songlines diplomacy** (Pillar 7): дипломатия через обмен песнями вместо вестфальских договоров.
- **Linguistic drift** (Pillar 8): имена и термины дрейфуют между изолированными группами.
- Денисовцы как NPC-пришельцы с востока (3-й вид, не играбелен ещё).

### post-v1.0 — Следующие эпохи

- Мезолит (модуль)
- Неолитическая революция (модуль)
- Бронзовый век (модуль)
- … и далее по эпохам

### Очень long-term

- Code-моды (DLL или встроенный скриптовый язык).
- Real-time lockstep multiplayer.
- Mobile-порт.
- Денисовцы как третий играбельный вид.
- Заселение остальных континентов как модули («Out of Africa», «Into the Americas»).
